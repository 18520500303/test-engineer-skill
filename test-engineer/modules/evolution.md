# 自进化模块

管理测试知识沉淀、知识库健康回顾、进化日志记录。

---

## 触发场景

| 触发词 | 场景 | 动作 |
|---|---|---|
| "回顾" / "review" | 知识库回顾 | 执行知识库健康检查 |
| "知识库健康" | 知识库回顾 | 同上 |
| "进化日志" | 查看日志 | 读取 `evolution-log.md` 展示变更历史 |

---

## 场景一：知识沉淀（自动触发）

**触发时机**：每个测试阶段完成后的"沉淀检查"环节（由 SKILL.md 自评审触发）。

**执行步骤**：

1. **识别可沉淀内容**：
   - 新发现的缺陷模式（如"某场景下必现的边界条件"）
   - 测试经验（如"某类需求的高效测试策略"）
   - 流程改进（如"某阶段的步骤可以优化"）

2. **检查去重**：
   - 读取 `../presets/{preset}/knowledge-base/` 对应目录
   - 与已有内容对比，只沉淀增量

3. **写入位置**：
   | 内容类型 | 沉淀位置 |
   |---|---|
   | 缺陷模式 | `../presets/{preset}/knowledge-base/defect-patterns/` |
   | 业务规则 | `../presets/{preset}/knowledge-base/business-rules/` |
   | 测试方法 | `../presets/{preset}/knowledge-base/methodologies/` |
   | 排查经验 | `../presets/{preset}/knowledge-base/troubleshoot-patterns/` |
   | 系统基线 | `../presets/{preset}/knowledge-base/system-baselines/` |

4. **追加进化日志**：记录沉淀了什么内容到 `evolution-log.md`

---

## 场景二：知识库回顾（手动触发）

**触发词**："回顾"、"知识库健康"

**执行步骤**：

1. **扫描知识库**：
   - 读取 `../presets/{preset}/knowledge-base/` 下所有子目录
   - 统计各目录文件数量和最后修改时间

2. **健康检查**：
   - [ ] **过时内容** — 是否有基于旧版本的知识（如已不存在的功能）？
   - [ ] **重复内容** — 不同目录下是否有相似/重复的知识文件？
   - [ ] **知识盲点** — 最近版本测试中发现的问题，是否已沉淀？
   - [ ] **覆盖度** — 各目录是否均衡？是否有长期未更新的领域？

3. **输出回顾报告**：
   ```
   ## 知识库健康报告 - {日期}
   
   ### 统计
   - 缺陷模式：X 个文件（最近更新：YYYY-MM-DD）
   - 业务规则：X 个文件（最近更新：YYYY-MM-DD）
   - ...
   
   ### 发现的问题
   - [ ] {问题描述} → 建议处理方式
   
   ### 建议
   - {建议}
   ```

4. **追加进化日志**：记录回顾结果

---

## 场景三：流程效率分析（版本归档时自动触发）

**触发时机**：版本归档完成后。

**执行步骤**：

1. **回顾本版本流程**：
   - 各阶段耗时（从状态文件时间戳推算）
   - 哪些需求类型用例质量高/低
   - 交叉复核发现的问题数量和类型

2. **对比历史版本**：
   - 与上一版本的效率对比
   - 是否有重复出现的流程卡点

3. **提出改进建议**：具体的、可执行的优化点

4. **追加进化日志**

---

## 进化日志格式

文件位置：`evolution-log.md`（与 SKILL.md 同级）

格式：
```markdown
## YYYY-MM-DD

- [沉淀] 新增缺陷模式：{描述}，写入 defect-patterns/xxx.md
- [回顾] 知识库健康检查：发现 X 个过时文件，已清理
- [流程] v{版本} 归档分析：需求阶段平均耗时下降，设计阶段交叉复核发现率提高
- [框架] SKILL.md 新增 xxx 功能
```

**日志分类标签**：
- `[沉淀]` — 新知识写入 knowledge-base
- `[回顾]` — 知识库健康检查结果
- `[流程]` — 流程效率分析
- `[框架]` — 框架自身变更（模块新增/修改/删除）

---

## 核心原则

1. **沉淀不沉没** — 每次测试都可能产生新知识，自评审环节确保不遗漏
2. **增量沉淀** — 先读已有内容，只写入新发现，避免重复
3. **可追溯** — 所有变更记录到 evolution-log.md，随时可查
4. **不阻塞流程** — 沉淀检查是轻量提醒，用户可选择跳过
