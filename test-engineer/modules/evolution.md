# 自进化模块

管理三层进化：知识沉淀（自动）、流程优化（自动分析）、**Prompt 层自修改（需人工审批）**。

---

## 触发场景

| 触发词 | 场景 | 动作 |
|---|---|---|
| "回顾" / "知识库健康" | 知识库回顾 | 执行知识库健康检查 |
| "进化日志" | 查看日志 | 读取 `evolution-log.md` 展示变更历史 |
| "进化" / "优化模块" | Prompt 层进化 | 分析信号 → 生成改进提案 |
| "进化信号" | 查看信号 | 展示 `evolution-signals.jsonl` 当前累积信号 |
| "回滚 {模块名}" | 回滚进化 | 展示该模块 git 历史，选择回滚点 |

---

## 一、信号收集（持续运行）

### 信号类型

| 信号类型 | 来源 | 采集时机 |
|---|---|---|
| `user_correction` | 用户否定模块输出、要求重做、手动修改结果 | 模块输出被否定时立即记录 |
| `pattern` | 同类需求反复在同一模块出现相同问题 | 归档时自动分析 |
| `bottleneck` | 某阶段耗时异常、反复修改次数过多 | 归档时从状态文件分析 |
| `knowledge` | 新的缺陷模式、测试经验 | 阶段完成后沉淀检查 |

### 信号存储

文件位置：`evolution-signals.jsonl`（与 SKILL.md 同级）

格式：
```jsonl
{"date": "2026-05-20", "type": "user_correction", "module": "test-case-generator", "context": "生成平台线用例时遗漏了权限相关场景", "version": "v1.1.29", "severity": "medium"}
{"date": "2026-05-21", "type": "pattern", "module": "algo-test-report", "context": "连续两个版本报告结论措辞被用户修改为更保守的表达", "version": "A2.27", "severity": "low"}
{"date": "2026-05-22", "type": "bottleneck", "module": "req-manager", "context": "需求润色阶段平均被修改3次才通过", "version": "v1.1.29", "severity": "high"}
```

### 采集集成点

SKILL.md 的自评审流程中，在"质量检查 + 沉淀检查"之后，新增**信号采集**步骤：

```
阶段完成 → 自评审 → 更新状态文件
                ↓
        质量检查 + 沉淀检查（现有）
                ↓
        信号采集（新增）：
        1. 本阶段模块输出是否被用户否定或修改？
           - 是 → 追加 user_correction 信号到 evolution-signals.jsonl
           - 记录：哪个模块、什么内容被否定、用户的修正方向
        2. 本阶段是否反复执行（≥2次）？
           - 是 → 追加 bottleneck 信号
```

## 二、知识沉淀（自动执行，现有机制保留）

**触发时机**：每个测试阶段完成后的"沉淀检查"环节。

**执行步骤**：

1. **识别可沉淀内容**：新缺陷模式、测试经验、流程改进
2. **检查去重**：读取 `{paths.knowledge_base}` 对应目录，只沉淀增量
3. **写入位置**：
   | 内容类型 | 沉淀位置 |
   |---|---|
   | 缺陷模式 | `{paths.knowledge_base}defect-patterns/` |
   | 业务规则 | `{paths.knowledge_base}business-rules/` |
   | 测试方法 | `{paths.shared_knowledge}testing-frameworks/` |
   | 排查经验 | `{paths.knowledge_base}diagnosis/` |
   | 系统基线 | `{paths.knowledge_base}system-baselines/` |
4. **追加进化日志** + **追加 knowledge 类型信号**

---

## 三、知识库回顾（手动触发，现有机制保留）

**触发词**："回顾"、"知识库健康"

**执行步骤**：

1. 扫描 `{paths.knowledge_base}` 下所有子目录
2. 健康检查：
   - [ ] 过时内容 — 基于旧版本的知识？
   - [ ] 重复内容 — 不同目录下相似/重复？
   - [ ] 知识盲点 — 最近版本测试中的问题是否已沉淀？
   - [ ] 覆盖度 — 各目录是否均衡？
3. 输出回顾报告
4. 追加进化日志

---

## 四、流程效率分析（版本归档时自动触发）

**触发时机**：版本归档完成后。

**执行步骤**：

1. 回顾本版本流程（各阶段耗时、用例质量、交叉复核发现）
2. 对比历史版本（效率对比、重复卡点）
3. 识别模式信号：
   - 某模块被修正 ≥2 次 → 追加 pattern 信号
   - 某阶段耗时显著增长 → 追加 bottleneck 信号
4. 提出改进建议
5. 追加进化日志

## 五、Prompt 层进化（核心新增能力）

### 触发条件

三个触发点，任一满足即可触发：

| 触发点 | 条件 | 类型 |
|---|---|---|
| 版本归档后 | 归档完成时自动检查信号 | 自动 |
| 信号累积阈值 | 同一模块累积 ≥3 条 user_correction 或 pattern 信号 | 自动提醒 |
| 手动触发 | 用户说"进化"/"优化模块" | 手动 |

### 决策流程

```
① 读取 evolution-signals.jsonl 中所有未处理信号
    ↓
② 按模块聚合：统计每个模块的信号数量和类型
    ↓
③ 选择问题最集中的模块（信号数最多 + severity 加权）
    ↓
④ 读取该模块当前 prompt（modules/{module}.md）
    ↓
⑤ 分析：结合信号上下文，定位 prompt 中哪段指令导致了问题
   - 只关注 <!-- evolution:mutable --> 标记的段落
   - 未标记的段落不允许修改
    ↓
⑥ 生成改进提案，格式如下：

   ---
   **进化提案**
   - 目标模块：`modules/test-case-generator.md`
   - 触发信号：3 条 user_correction（v1.1.28×1, v1.1.29×2），均涉及权限场景遗漏
   - 修改位置：场景覆盖清单（第 XX 行附近）
   - 当前内容：
     ```
     每个功能点必须覆盖：正向流程 + 边界值 + 异常场景
     ```
   - 建议改为：
     ```
     每个功能点必须覆盖：正向流程 + 边界值 + 异常场景 + 权限控制（含无权限/低权限/跨角色）
     ```
   - 预期效果：减少权限相关用例遗漏
   ---

    ↓
⑦ 展示提案给用户，等待审批
    ↓
⑧ 用户决策：
   ├→ 批准 → 执行修改 → git commit → 记录 evolution-log.md → 标记信号为已处理
   ├→ 部分批准 → 只改批准的部分
   └→ 拒绝 → 记录拒绝原因到 evolution-log.md（避免重复提议）
       拒绝记录格式：{"module": "xxx", "proposal_hash": "简述", "reject_reason": "...", "date": "..."}
```

### 可修改范围（白名单机制）

**只有被 `<!-- evolution:mutable -->` ... `<!-- evolution:end -->` 标记的段落才允许进化修改。**

| 可修改（标记为 mutable） | 不可修改（永远不标记） |
|---|---|
| 场景覆盖清单、检查项列表 | 触发词定义 |
| 模板中的措辞和格式要求 | 数据 Schema（由 shared/definitions.md 管控） |
| 默认值和阈值 | 接口定义（由 shared/interfaces.md 管控） |
| 质量标准描述 | 文件输出路径 |
| 具体的操作指令文本 | 模块间调用关系 |

### 信号处理后清理

已处理的信号从 `evolution-signals.jsonl` 中标记为 `"processed": true`，不删除（保留审计轨迹）。

下次进化分析时跳过 `processed: true` 的信号。

## 六、进化日志格式

文件位置：`evolution-log.md`（与 SKILL.md 同级）

```markdown
## YYYY-MM-DD

- [沉淀] 新增缺陷模式：{描述}，写入 defect-patterns/xxx.md
- [回顾] 知识库健康检查：发现 X 个过时文件，已清理
- [流程] v{版本} 归档分析：需求阶段平均耗时下降，设计阶段交叉复核发现率提高
- [prompt] {模块名}.md：{修改简述}
  - 触发信号：{信号数量和类型}
  - 审批状态：已批准 / 已拒绝（原因：{原因}）
- [框架] SKILL.md 新增 xxx 功能
```

**日志分类标签**：
- `[沉淀]` — 新知识写入 knowledge-base
- `[回顾]` — 知识库健康检查结果
- `[流程]` — 流程效率分析
- `[prompt]` — Prompt 层进化（新增）
- `[prompt:rejected]` — 被拒绝的进化提案（新增）
- `[框架]` — 框架自身变更

---

## 核心原则

1. **宪法不动** — SKILL.md 路由逻辑、shared/ 数据契约和接口永远不被自动修改
2. **标记即边界** — 只有 `<!-- evolution:mutable -->` 标记的段落才允许进化修改
3. **人工审批** — 所有 Prompt 修改必须展示 diff 给用户确认后才生效
4. **拒绝留痕** — 被拒绝的提案记录原因，避免重复提议相同的改动
5. **可回滚** — 每次修改有 git commit，用户可说"回滚 {模块名}"恢复
6. **增量进化** — 每次只改一个模块的一个方面，不做大重构
7. **沉淀不沉没** — 知识沉淀是自动的，不阻塞流程
